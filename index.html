Campus Safety App - Project Overview

Introduction

I have been employed as a student officer at the Campus Safety Office at Lawrence University for a year now. Campus Safety's mission is to provide each student, faculty, and staff member with a safe and secure living, learning, and working environment. Every day, a variety of requests come in from students, faculty, and staff.

Recognizing the Problem

The problem we recognized was that requests were made via phone calls. Because phone conversations are unstructured, they take more time and can lead to miscommunication regarding the request details. Both the requester and the officer relied on their memory, and officers often wrote details down on paper to write up a report about the request later on. Additionally, communicating updates was difficult. For example, if a student requested a ride to the ER and an officer accepted via phone call, updating the student on the officer's arrival required looking up their phone number and manually sending a message or making another call.

Seeing these inefficiencies, my boss, the Campus Safety Director, gave me the opportunity to partner with the university to develop a mobile app for Campus Safety. This led to the creation of the Campus Safety App, which modernizes request handling through a structured, digital process.

The Solution

As a solution, I created a full stack mobile application. I made two versions of the app: a student version and an officer version. In the app, there are forms with designated fields for all types of requests, making them clear and structured. The request history is automatically saved in the database, so officers no longer have to struggle with reports.

What is a Full-Stack Application?

A full-stack app can be best understood with a restaurant analogy:

  • Frontend - The Waiter and Menu: This is what users interact with. Just like a waiter takes an order and brings it to the kitchen, the frontend collects user input and sends it to the backend.
  • Backend - The Chef and Kitchen: This is where the logic happens. Just like a chef takes the order from the waiter and prepares the food, the backend processes requests, performs necessary calculations, and sends responses.
  • Database - The Storage and Ingredients: This stores all necessary information. Just like a restaurant kitchen has a pantry storing ingredients, a database holds all user data, request history, and other information.

Full-Stack architecture

I chose React Native for my frontend because it allows one codebase to generate both iOS and Android apps, avoiding the need to learn Swift and Kotlin separately. My backend is built with Spring Boot, and my database is a MySQL database. I deployed my service on AWS.

Database Structure

The Campus Safety App's backend uses a Spring Boot JPA setup with a MySQL database. Here is the database schema:

  • Users Table:
    • userID (UUID, Primary Key)
    • username, password, usertype
    • firstname, lastname, phone, studentID
    • verified, verificationCode, verificationCodeExpiry
  • Requests Table:
    • requestID (Integer, Primary Key)
    • requesterID (FK → users), receiverID (FK → users)
    • requestType, location, message
    • requestStatus (pending, accepted, completed, canceled)
    • reserved, reservationDue
  • ChatMessages Table:
    • chatMessageID (Integer, Primary Key)
    • senderID (FK → users), receiverID (FK → users)
    • messageContent, messageTimestamp
  • RequestOptions Table:
    • requestOptionID (Integer, Primary Key)
    • requestTitle (String)
  • LocationOptions Table:
    • locationOptionID (Integer, Primary Key)
    • locationName (String)
  • OfficerDriverOptions Table:
    • OfficerDriverOptionID (Integer, Primary Key)
    • firstname, lastname, type (officer/driver)
  • RefreshToken Table:
    • id (Integer, Primary Key)
    • token, expiryDate
    • userID (FK → users)

This schema supports the app’s core features, including authentication, request handling, messaging, and role-based access control.

Backend Implementation

The backend of the Campus Safety App is built using Spring Boot and follows a structured REST API architecture. It is designed for scalability and high availability, ensuring smooth operations for students and officers using the mobile app.

  • Spring Boot Framework - Handles API requests and business logic.
  • JPA (Java Persistence API) - Used for efficient database operations.
  • MySQL Database - Stores user information, requests, chat messages, and tokens.
  • Spring Security - Manages authentication and role-based access control.
  • JWT (JSON Web Token) Authentication - Ensures secure user sessions.
  • Email Service - Sends automated verification emails using JavaMailSender.
  • AWS Hosting - The backend is deployed on AWS EC2, with load balancing and auto-scaling enabled.

Security Implementation in Detail

JWT Authentication

  • Uses JSON Web Tokens (JWT) to authenticate users.
  • Tokens are signed using HMAC SHA-256 for security.
  • Each user gets a 15-minute Access Token and a 3-month Refresh Token.
  • Security Flow:
    • User logs in → Receives an Access Token.
    • Access Token expires → Uses Refresh Token to get a new one.
    • Refresh Token expires → User must log in again.

Refresh Tokens

  • Stored in the database (`RefreshToken` table) for better security.
  • Can be used to re-authenticate users without needing a password.
  • If the refresh token is expired, the user must re-login.

📧 Automated Email Verification

  • When a user registers, a 6-digit verification code is generated.
  • The code is valid for 15 minutes.
  • The system sends an email with the code using `JavaMailSender`.
  • User enters the code → The account is activated.
  • If the code expires, the user can request a new one.

Secure API Endpoints

  • Public endpoints: `/user/register`, `/user/login`, `/user/refreshToken`.
  • All other API endpoints require JWT authentication.
  • Spring Security automatically rejects unauthorized requests.

Why AWS?

AWS provides the scalability, security, and reliability required for an application that needs to handle real-time requests and dynamic workloads. This setup ensures that the system remains responsive even during peak request times, with automatic scaling and efficient load balancing.

AWS Infrastructure

The backend of the Campus Safety App is hosted on AWS using a robust, scalable architecture.

The system architecture follows this workflow:

  • The client (mobile app) communicates with an Application Load Balancer (ALB), which distributes incoming traffic.
  • The ALB checks its Target Group and routes requests to the available EC2 instances.
  • The EC2 instances process requests and interact with an RDS instance that stores all request and user data in a MySQL database.
  • An Auto Scaling Group (ASG) manages the EC2 instances, automatically launching or terminating them based on demand.
  • When the ASG launches a new EC2 instance, it uses a Launch Template to configure the server properly.
  • An EventBridge Rule detects when a new EC2 instance is created and triggers a Lambda function that adds the new instance to the ALB's Target Group.
